<overview>
## Problem Statement

The current Butabot Agent implementation consists of ~2,250 lines of custom code that is tightly coupled to Slack, making multi-platform support difficult. The manual orchestration code (tool loops, history management) is error-prone and hard to maintain. Adding new platforms requires significant code duplication, and there's a high maintenance burden with custom implementations of framework-level concerns.

**Specific Pain Points:**
- Current implementation is Slack-only (not multi-platform)
- Manual tool execution loop (~570 lines) is complex and error-prone
- Custom MCP client wrapper (~400 lines) duplicates framework functionality
- Tool approval workflow (~370 lines) is tightly coupled to Slack
- Adding new platforms requires rewriting core logic
- High maintenance burden with custom orchestration code

**Why Existing Solutions Don't Work:**
- Current custom implementation doesn't scale to multiple platforms
- Manual orchestration code is brittle and hard to test
- Framework-level concerns (tool loops, history management) should be handled by a framework
- No clear separation between platform logic and core agent logic

## Target Users

### Primary Persona: Drupal Developer
- **Needs:** AI assistant for Drupal development tasks (file operations, module management, code generation)
- **Platform:** Primarily Slack (workplace), occasionally Discord (community)
- **Workflows:** File operations, code generation, Drupal-specific commands
- **Pain Points:** Waiting for tool approvals, unclear what bot is doing, platform lock-in

### Secondary Persona: General Developer
- **Needs:** General development assistance (web search, file operations, code analysis)
- **Platform:** Slack or Discord (varies by team)
- **Workflows:** Web search, code analysis, file operations
- **Pain Points:** Multi-platform inconsistency, lack of transparency, need to use different bots per platform

## Success Metrics

**Quantifiable Outcomes:**
- **Code Reduction:** 75% reduction (2,250 → 500 lines of custom code)
- **Platform Addition Time:** <1 day per new platform
- **Feature Parity:** 100% of existing features maintained
- **Performance:** Response times ≤ current implementation
- **Test Coverage:** ≥80% code coverage
- **Deployment:** Zero downtime migration from old to new implementation

**Measurable Improvements:**
- Multi-platform support: 2+ platforms (Slack + Discord) working simultaneously
- Tool approval workflow: 100% of dangerous operations require approval
- Error handling: User-friendly error messages for 100% of error cases
- Maintainability: Framework updates improve bot automatically (no custom code changes needed)
</overview>

<functional-decomposition>
## Capability Tree

### Capability: Platform Integration
Handles communication with chat platforms (Slack, Discord, future platforms), translating platform-specific events into standardized format and routing responses back.

#### Feature: Message Reception
- **Description**: Receive and normalize messages from chat platforms (Slack mentions, Discord messages)
- **Inputs**: Platform-specific event objects (Slack Bolt events, Discord.py events)
- **Outputs**: Standardized `PlatformMessage` object with thread_id, user_id, content, platform, channel_id, is_mention
- **Behavior**: Extract message text, user ID, channel ID, thread ID from platform event. Normalize to common format. Detect if bot was mentioned.

#### Feature: Message Transmission
- **Description**: Send messages back to chat platforms in platform-appropriate format
- **Inputs**: Thread ID, message content (text or structured blocks), message type (response, status, error)
- **Outputs**: Platform-specific message sent to channel/thread
- **Behavior**: Format content for platform (Slack blocks, Discord embeds). Post to correct channel/thread. Handle platform-specific formatting requirements.

#### Feature: Approval Request Handling
- **Description**: Request tool approval from users via platform-specific interactive components (buttons)
- **Inputs**: Thread ID, tool name, tool parameters, approval timeout
- **Outputs**: Boolean (approved/denied), approval request message posted to platform
- **Behavior**: Create approval request with tool details. Post interactive buttons to platform. Wait for user response (async). Handle timeout (default deny). Return approval decision.

#### Feature: Button Interaction Processing
- **Description**: Process user button clicks (approve/deny) from platforms
- **Inputs**: Platform-specific button interaction event, approval ID
- **Outputs**: Approval decision resolved, approval tracker updated
- **Behavior**: Extract approval ID from button click. Resolve pending approval request. Update approval tracker. Continue tool execution or block based on decision.

#### Feature: Factoid Checking
- **Description**: Check incoming messages against factoid configuration before agent processing
- **Inputs**: Message text, factoid configuration (JSON), cooldown tracker
- **Outputs**: Factoid response string (if matched) or None, cooldown updated
- **Behavior**: Match message text against factoid triggers (exact match, case-sensitive). Check cooldown (30 seconds global). Select random response if multiple available. Return response or None.

### Capability: Tool Approval Workflow
Provides user control over AI tool execution through interactive approval requests with async waiting and exception-based blocking.

#### Feature: Approval Rule Management
- **Description**: Load and query tool-specific approval requirements from configuration
- **Inputs**: Tool name, approval rules configuration (YAML)
- **Outputs**: Boolean indicating if tool requires approval
- **Behavior**: Load approval rules from `approval_rules.yaml`. Query rules by tool name. Return default behavior if tool not specified. Support tool category rules.

#### Feature: Tool Usage Reporting
- **Description**: Report tool usage to users in real-time before execution
- **Inputs**: Thread ID, tool name, tool parameters, platform interface
- **Outputs**: Status message posted to platform
- **Behavior**: Format tool information (name, parameters). Send status update to platform via connector. Use platform-appropriate formatting (context blocks, embeds).

#### Feature: Approval Request Processing
- **Description**: Request approval from user, wait for response, block execution if denied
- **Inputs**: Thread ID, tool name, tool parameters, platform interface, approval rules
- **Outputs**: Boolean (approved/denied), approval request posted
- **Behavior**: Check if tool requires approval. If yes, request approval via platform (async). Wait for user response with timeout. Block execution by raising exception if denied. Continue if approved.

#### Feature: Tool Result Reporting
- **Description**: Report tool execution results to users after completion
- **Inputs**: Thread ID, tool name, tool result, success/error status, platform interface
- **Outputs**: Result message posted to platform, approval message updated
- **Behavior**: Format tool result (truncate if large). Send status update to platform. Update approval request message with result. Handle errors gracefully.

#### Feature: Approval Tracking
- **Description**: Track pending approval requests across concurrent conversations
- **Inputs**: Approval ID, thread ID, tool use ID, approval decision
- **Outputs**: Approval state tracked, approval requests resolved
- **Behavior**: Store pending approvals by ID. Map tool_use_id to approval_id. Resolve approvals when user responds. Clean up after completion or timeout.

### Capability: Agent Orchestration
Manages AI agent interactions, tool execution, and conversation flow using fast-agent framework.

#### Feature: Agent Definition
- **Description**: Define AI agent using fast-agent framework with MCP servers and configuration
- **Inputs**: Agent configuration (model, MCP servers, parameters), fast-agent framework
- **Outputs**: Configured agent instance ready for message processing
- **Behavior**: Use `@fast.agent` decorator to define agent. Configure MCP servers from `fastagent.config.yaml`. Set model and parameters. Initialize framework.

#### Feature: Message Processing
- **Description**: Process user messages through agent and return responses
- **Inputs**: Thread ID, user message text, agent instance, conversation history
- **Outputs**: Agent response (text, tool calls, multi-modal content)
- **Behavior**: Send message to agent via `agent.send()`. Framework handles tool execution loop automatically. Return response with text and tool results. Maintain conversation history per thread.

#### Feature: Tool Execution Loop
- **Description**: Automatically execute tools requested by agent, handle results, continue conversation
- **Inputs**: Tool calls from agent, tool definitions from MCP servers, approval workflow
- **Outputs**: Tool execution results, continued agent response
- **Behavior**: Framework detects tool calls. Execute tools via MCP. Handle tool results. Continue conversation with results. Repeat until agent completes (no more tool calls).

#### Feature: Conversation History Management
- **Description**: Maintain separate conversation contexts per thread/conversation
- **Inputs**: Thread ID, messages (user and assistant), framework
- **Outputs**: Conversation history stored and retrieved per thread
- **Behavior**: Framework manages history per thread ID. Store full message arrays. Retrieve history for context. Isolate conversations across threads.

#### Feature: Multi-Modal Content Support
- **Description**: Handle files, images, PDFs in messages and tool results
- **Inputs**: Multi-modal content (files, images), platform interface, tool results
- **Outputs**: Content processed and displayed appropriately
- **Behavior**: Framework handles multi-modal content automatically. Detect image results from tools. Upload images to platform. Display content in platform-appropriate format.

### Capability: MCP Tool Integration
Manages integration with MCP (Model Context Protocol) servers for tool discovery and execution.

#### Feature: Tool Discovery
- **Description**: Automatically discover tools from configured MCP servers
- **Inputs**: MCP server configurations, fast-agent framework
- **Outputs**: List of available tools with schemas
- **Behavior**: Framework connects to MCP servers. Discovers tools automatically. Converts schemas (MCP → Anthropic format). Caches tool definitions.

#### Feature: Tool Execution
- **Description**: Execute tools from MCP servers with proper parameter handling
- **Inputs**: Tool name, tool parameters, MCP server connection
- **Outputs**: Tool execution result (structured data, text, errors)
- **Behavior**: Framework routes tool calls to correct MCP server. Executes tool with parameters. Handles errors gracefully. Returns structured results.

#### Feature: Multi-Server Management
- **Description**: Manage multiple MCP servers (5-6 servers, 90+ tools) simultaneously
- **Inputs**: Multiple MCP server configurations, framework
- **Outputs**: Unified tool interface across all servers
- **Behavior**: Framework handles multiple servers automatically. Routes tool calls to correct server. Maintains connections. Handles server failures gracefully.

### Capability: Error Handling & User Feedback
Detects and handles errors gracefully, providing user-friendly messages while logging detailed information.

#### Feature: Error Detection
- **Description**: Detect API errors, rate limits, and tool failures
- **Inputs**: API responses, tool results, error messages
- **Outputs**: Error type identified, user-friendly message formatted
- **Behavior**: Check for common error patterns (529 overloaded, 429 rate limit, 401 auth, 500 internal). Extract error details. Format user-friendly message.

#### Feature: Error Reporting
- **Description**: Send user-friendly error messages to platforms while logging details
- **Inputs**: Error type, error details, thread ID, platform interface
- **Outputs**: User-friendly error message posted, detailed error logged
- **Behavior**: Format error for user (no technical jargon). Post to platform. Log detailed error for debugging. Continue operation if possible.

#### Feature: Error Recovery
- **Description**: Handle errors gracefully, allow bot to continue operating
- **Inputs**: Error context, system state
- **Outputs**: System continues operating, error handled
- **Behavior**: Catch and handle errors at appropriate levels. Don't crash on single tool failure. Provide fallback behavior. Log errors for debugging.

### Capability: Configuration Management
Manages configuration files, environment variables, and secrets for the bot.

#### Feature: Configuration Loading
- **Description**: Load configuration from YAML files and environment variables
- **Inputs**: Configuration files (`fastagent.config.yaml`, `approval_rules.yaml`), environment variables
- **Outputs**: Loaded configuration objects
- **Behavior**: Parse YAML files. Substitute environment variables (${VAR_NAME}). Validate configuration. Provide defaults for missing values.

#### Feature: Secret Management
- **Description**: Manage API keys and credentials securely
- **Inputs**: `fastagent.secrets.yaml` file, environment variables
- **Outputs**: Secrets available to components that need them
- **Behavior**: Load secrets from gitignored file. Substitute environment variables. Provide secrets to framework and connectors. Never commit secrets to version control.
</functional-decomposition>

## Module Definitions

### Module: connectors
- **Maps to capability**: Platform Integration
- **Responsibility**: Handle platform-specific communication (Slack, Discord, future platforms)
- **File structure**:
  ```
  connectors/
  ├── interface.py           # PlatformInterface abstract base class
  ├── slack_connector.py     # SlackConnector (message reception, transmission, approval)
  ├── discord_connector.py   # DiscordConnector (message reception, transmission, approval)
  └── __init__.py            # Public exports
  ```
- **Exports**:
  - `PlatformInterface` - Abstract base class for platform connectors
  - `SlackConnector` - Slack platform implementation
  - `DiscordConnector` - Discord platform implementation
  - `PlatformMessage` - Standardized message data model

### Module: agent
- **Maps to capability**: Agent Orchestration
- **Responsibility**: Manage AI agent interactions using fast-agent framework
- **File structure**:
  ```
  agent/
  ├── tool_agent.py          # ChatPlatformToolAgent with hooks
  ├── agent_config.py        # Agent definition (@fast.agent)
  └── __init__.py            # Public exports
  ```
- **Exports**:
  - `ChatPlatformToolAgent` - Custom agent class with ToolRunnerHooks
  - `create_agent()` - Factory function to create configured agent instance

### Module: approval
- **Maps to capability**: Tool Approval Workflow
- **Responsibility**: Manage tool approval rules and tracking
- **File structure**:
  ```
  approval/
  ├── rules_manager.py       # ApprovalRulesManager (loads and queries rules)
  ├── approval_tracker.py    # Approval tracking system
  └── __init__.py            # Public exports
  ```
- **Exports**:
  - `ApprovalRulesManager` - Loads and queries approval rules
  - `ApprovalTracker` - Tracks pending approvals
  - `ToolApprovalDenied` - Exception for denied approvals

### Module: factoids
- **Maps to capability**: Platform Integration (Factoid Checking feature)
- **Responsibility**: Simple string-triggered responses without Claude
- **File structure**:
  ```
  factoids.py                # Existing file, minimal changes
  ```
- **Exports**:
  - `FactoidManager` - Checks factoids before agent processing (existing)

### Module: config
- **Maps to capability**: Configuration Management
- **Responsibility**: Configuration files and loading
- **File structure**:
  ```
  config/
  ├── fastagent.config.yaml  # fast-agent configuration
  ├── fastagent.secrets.yaml # Secrets (gitignored)
  └── approval_rules.yaml    # Approval rules
  ```
- **Note**: Configuration files, not Python modules. Loaded by other modules.

<dependency-graph>
## Dependency Chain

### Foundation Layer (Phase 0)
No dependencies - these are built first.

- **config**: Configuration file structure and loading utilities
  - Provides: Configuration loading, secret management
  - No dependencies

- **connectors.interface**: Platform abstraction interface
  - Provides: `PlatformInterface` abstract base class, `PlatformMessage` data model
  - No dependencies

### Framework Integration Layer (Phase 1)
Depends on foundation layer.

- **agent.agent_config**: fast-agent framework integration
  - Provides: Agent definition using `@fast.agent`, MCP server configuration
  - Depends on: [config]
  - Framework dependency: fast-agent-mcp package

### Tool Flow Layer (Phase 2)
Depends on foundation and framework layers.

- **approval.rules_manager**: Approval rules management
  - Provides: Load and query approval rules from YAML
  - Depends on: [config]

- **approval.approval_tracker**: Approval tracking system
  - Provides: Track pending approvals, resolve approvals
  - Depends on: [approval.rules_manager]

- **agent.tool_agent**: Custom agent with ToolRunnerHooks
  - Provides: `ChatPlatformToolAgent` class with approval hooks
  - Depends on: [agent.agent_config, connectors.interface, approval.rules_manager, approval.approval_tracker]
  - Framework dependency: fast-agent ToolRunnerHooks

### Platform Layer (Phase 3)
Depends on foundation, framework, and tool flow layers.

- **connectors.slack_connector**: Slack platform implementation
  - Provides: Slack message handling, approval requests, button interactions
  - Depends on: [connectors.interface, agent.tool_agent, factoids]
  - External dependency: slack-bolt library

- **connectors.discord_connector**: Discord platform implementation
  - Provides: Discord message handling, approval requests, button interactions
  - Depends on: [connectors.interface, agent.tool_agent, factoids]
  - External dependency: discord.py library

### Integration Layer (Phase 4)
Depends on all previous layers.

- **factoids**: Factoid system integration
  - Provides: Factoid checking before agent processing
  - Depends on: [connectors.slack_connector, connectors.discord_connector]
  - Note: Existing module, minimal changes needed

- **app**: Main application entry point
  - Provides: Application startup, event loop, platform connector initialization
  - Depends on: [connectors.slack_connector, connectors.discord_connector, agent.tool_agent]
</dependency-graph>

## Critical Test Scenarios

### Module: connectors.slack_connector
**Happy path**:
- Receive Slack mention event → Normalize to PlatformMessage → Forward to agent → Format response → Post to Slack
- Expected: Message processed correctly, response posted to correct thread

**Edge cases**:
- Message in thread vs channel → Handle thread_ts correctly
- Bot mentioned vs not mentioned → Detect is_mention correctly
- Empty message → Skip processing
- Expected: All edge cases handled gracefully

**Error cases**:
- Slack API error → Catch and log, send user-friendly message
- Network timeout → Retry or fail gracefully
- Expected: Errors don't crash bot, user sees helpful message

**Integration points**:
- Factoid checking before agent → Factoid matched, response sent, agent not called
- Approval request → Button posted, user clicks, approval resolved
- Expected: All integrations work correctly

**Edge cases**:
- Direct message vs channel message → Handle context correctly
- Bot mentioned vs not mentioned → Detect mention correctly
- Expected: All edge cases handled gracefully

**Error cases**:
- Discord API error → Catch and log, send user-friendly message
- Network timeout → Retry or fail gracefully

**Integration points**:

- Expected: Errors don't crash system, operation continues

**Integration points**:
- agent.tool_agent → Approvals tracked for tool calls

- Expected: Integration works, approvals tracked and resolved correctly

## Technology Stack

**Languages & Frameworks:**
- Python 3.11+ (existing)
- fast-agent-mcp (new framework)
- slack-bolt (existing)
- discord.py (new library)

**Key Libraries:**
- fast-agent-mcp: Agent orchestration framework
- slack-bolt: Slack platform integration
- discord.py: Discord platform integration
- PyYAML: Configuration file parsing
- python-dotenv: Environment variable management

**Decision: fast-agent Framework**
- **Rationale**: Handles AI orchestration, tool execution loops, conversation history, MCP integration automatically. Reduces custom code by 75%.
- **Trade-offs**: Dependency on external framework, learning curve, framework stability risk
- **Alternatives considered**: 
  - Keep current custom implementation (rejected: too much code, not multi-platform)
  - Use Anthropic Agent SDK (rejected: doesn't support MCP, less control)
  - Hybrid approach (fallback: extract connectors, keep current orchestration)

**Decision: Platform Abstraction Interface**
- **Rationale**: Enables multi-platform support with minimal code duplication. Clear separation of concerns.
- **Trade-offs**: Additional abstraction layer, slight overhead
- **Alternatives considered**:
  - Platform-specific implementations (rejected: code duplication)
  - Generic webhook approach (rejected: loses platform-specific features like buttons)

**Decision: YAML Configuration Files**
- **Rationale**: Human-readable, version-controllable, supports environment variable substitution
- **Trade-offs**: Requires YAML parsing, less type-safe than Python config
- **Alternatives considered**:
  - Python config files (rejected: harder to edit, secrets in code)
  - JSON config (rejected: less readable, no comments)
  - Environment variables only (rejected: too many variables, hard to manage)
