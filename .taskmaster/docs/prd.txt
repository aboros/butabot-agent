# Overview  
The Butabot Agent Refactor project transforms the existing custom-built Slack bot into a multi-platform AI agent using the fast-agent framework. Currently, the bot consists of ~2,250 lines of custom code that manually handles tool execution, conversation management, and platform integration. This refactor will reduce the codebase by 75% (to ~500 lines) while enabling support for multiple chat platforms (Slack, Discord, and future platforms) and maintaining 100% feature parity.

**Problem Statement:**
- Current implementation is tightly coupled to Slack, making multi-platform support difficult
- Manual orchestration code (tool loops, history management) is error-prone and hard to maintain
- Adding new platforms requires significant code duplication
- High maintenance burden with custom implementations of framework-level concerns

**Solution:**
Migrate to fast-agent framework which handles:
- AI orchestration and tool execution loops
- MCP server integration and tool management
- Conversation history management
- Multi-modal content support
- Error handling and retry logic

**Value Proposition:**
- 75% code reduction (2,250 → 500 lines)
- Multi-platform support in <1 day per platform
- Framework-tested orchestration logic
- Reduced maintenance burden
- Future-proof architecture with workflow patterns (chaining, parallelism)

**Target Users:**
- Development teams using the bot for Drupal development
- Users across Slack and Discord platforms
- Future users on Teams, WhatsApp, Telegram, etc.

# Core Features  

## 1. Multi-Platform Support
**What it does:** Enables the bot to operate identically across Slack, Discord, and future platforms through platform-agnostic connectors.

**Why it's important:** Expands bot reach without code duplication. Each platform gets full feature parity with minimal platform-specific code (~50 lines per platform).

**How it works:** Platform connectors translate platform-specific events (Slack mentions, Discord messages) into standardized message format, forward to shared agent core, and route responses back in platform-specific format.

## 2. Tool Approval Workflow
**What it does:** Provides user control over AI tool execution through interactive approval requests with async waiting and exception-based blocking.

**Why it's important:** Safety and transparency. Users see what tools are being used and can approve/deny dangerous operations (file deletion, git pushes, etc.) before execution.

**How it works:** Custom agent class implements ToolRunnerHooks that intercept tool calls, check approval rules (YAML config), request approval from user (async), wait for response, and block execution via exception if denied. Results are reported back to user.

## 3. MCP Tool Integration
**What it does:** Automatically discovers and manages 90+ tools from 5-6 MCP servers (filesystem, Drupal, Brave Search, etc.) without manual tool registration.

**Why it's important:** Access to rich tool ecosystem. Framework handles tool discovery, schema conversion, and execution automatically.

**How it works:** fast-agent framework automatically discovers tools from configured MCP servers, converts schemas, and provides them to the agent. Custom code only needs to handle approval workflow.

## 4. Thread-Aware Conversations
**What it does:** Maintains separate conversation contexts per thread/conversation, enabling natural multi-turn dialogues.

**Why it's important:** User experience. Users can have multiple conversations simultaneously without context mixing.

**How it works:** Framework manages conversation history per thread ID. Each platform connector provides thread/conversation identifiers, and framework maintains isolated message histories.

## 5. Factoid System
**What it does:** Provides simple string-triggered responses for common queries (e.g., "ping" → "pong") without invoking Claude.

**Why it's important:** Performance and user experience. Instant responses for common queries, reduces API costs, prevents unnecessary Claude calls.

**How it works:** Platform connectors check factoid configuration before forwarding to agent. If match found, respond immediately and skip Claude processing. Supports cooldowns, multiple responses, and mention-only triggers.

## 6. Image Generation & Handling
**What it does:** Supports image generation tools and posts images directly to chat platforms.

**Why it's important:** Rich content support. Users can generate images and see them inline in chat.

**How it works:** Tool results containing images are detected, uploaded to platform using platform-specific APIs, and posted to conversation thread.

## 7. Error Handling & User Feedback
**What it does:** Detects API errors, rate limits, and tool failures, presenting user-friendly messages while logging detailed errors.

**Why it's important:** User experience and debugging. Users see helpful messages, developers get detailed logs.

**How it works:** Framework and custom hooks detect errors, format user-friendly messages, send to platform, and log detailed information for debugging.

# Technical Architecture  

## System Components

### 1. Platform Connectors Layer
**Purpose:** Translate platform-specific events into standardized format and route responses back.

**Components:**
- `SlackConnector` (~50-100 lines)
  - Handles Slack Bolt events (`app_mention`, `message`)
  - Handles button interactions (`tool_approve`, `tool_deny`)
  - Normalizes messages to `PlatformMessage` format
  - Routes responses back to Slack with proper formatting
  - Handles factoid checking before agent processing

- `DiscordConnector` (~50-100 lines)
  - Handles Discord.py events (message events)
  - Handles button interactions (approve/deny)
  - Normalizes messages to `PlatformMessage` format
  - Routes responses back to Discord with proper formatting
  - Handles factoid checking before agent processing

**Interface:**
```python
class PlatformInterface:
    async def send_message(self, thread_id: str, content: str) -> None
    async def send_status(self, thread_id: str, status: str) -> None
    async def request_approval(self, thread_id: str, tool_name: str, params: dict) -> bool
    async def send_error(self, thread_id: str, error: str) -> None
```

### 2. Tool Flow Orchestration Layer
**Purpose:** Provide full visibility and control over AI tool execution with approval workflows.

**Component:**
- `ChatPlatformToolAgent` (~200-300 lines)
  - Extends fast-agent's `ToolAgent`
  - Implements `ToolRunnerHookCapable` interface
  - Provides `ToolRunnerHooks` implementation:
    - `before_tool_call`: Approval workflow, tool reporting
    - `after_tool_call`: Result reporting, error handling
  - Manages approval rules from YAML config
  - Handles async approval waiting
  - Blocks tool execution via exceptions

**Key Methods:**
```python
async def _handle_tool_approval(self, runner, tool_call):
    # Report tool usage
    # Check approval rules
    # Request approval (async)
    # Block if denied (raise exception)

async def _report_tool_result(self, runner, message):
    # Report tool results
    # Handle errors
    # Update approval messages
```

### 3. fast-agent Framework Core
**Purpose:** Handle all AI orchestration, MCP tool management, and conversation logic.

**Components:**
- Agent definition via `@fast.agent` decorator
- MCP server integration (automatic tool discovery)
- Tool execution loop (automatic)
- Conversation history management (automatic)
- Multi-modal content support (automatic)
- Error handling and retry logic (automatic)

**Configuration:**
- `fastagent.config.yaml`: Model settings, MCP servers, agent parameters
- `fastagent.secrets.yaml`: API keys, credentials (gitignored)

### 4. Approval Rules System
**Purpose:** Configurable tool approval requirements.

**Component:**
- `approval_rules.yaml`: Tool-specific approval configuration
- `ApprovalRulesManager`: Loads and queries approval rules

**Format:**
```yaml
tools:
  filesystem.delete_directory:
    requires_approval: true
  filesystem.write_file:
    requires_approval: true
  filesystem.read_file:
    requires_approval: false
```

### 5. Factoid System
**Purpose:** Simple string-triggered responses without Claude.

**Component:**
- `factoids.json`: Factoid configuration (existing)
- `FactoidManager`: Checks factoids before agent processing (existing, minimal changes)

# Risks and Mitigations  

## Technical Challenges

### Risk 1: fast-agent Framework Limitations
**Challenge:** fast-agent may not support all required features (async hooks, tool blocking, MCP integration).

**Likelihood:** Medium
**Impact:** High - Could block entire refactor

**Mitigation:**
- Phase 0 verification before committing to refactor
- Create minimal POC to validate all critical features
- Have fallback plan: Hybrid approach (extract connectors, keep current orchestration)
- Monitor fast-agent development and community

**Contingency:** If framework doesn't support required features, use hybrid approach:
- Extract platform connectors (still valuable)
- Keep current orchestration code
- Add hook-like interface for tool approval
- Get multi-platform support without full framework dependency

### Risk 2: Tool Approval Workflow Complexity
**Challenge:** Async approval waiting in hooks may be complex or not supported.

**Likelihood:** Low
**Impact:** High - Core feature

**Mitigation:**
- Verify hook async support in Phase 0
- Test approval workflow thoroughly in POC
- Use asyncio.Future for approval tracking
- Implement timeout handling
- Test concurrent approval scenarios

**Contingency:** If hooks don't support async waiting:
- Use synchronous approval (blocking, less ideal)
- Or implement approval outside hooks (pre-process tool calls)

### Risk 3: MCP Integration Issues
**Challenge:** fast-agent may not integrate well with existing MCP servers or FastMCP setup.

**Likelihood:** Low
**Impact:** Medium - Could require MCP server changes

**Mitigation:**
- Test MCP integration early in Phase 0
- Verify all existing MCP servers work
- Test persistent connections (STDIO servers)
- Maintain FastMCP wrapper if needed

**Contingency:** If MCP integration issues:
- Keep FastMCP wrapper as adapter
- Or configure MCP servers differently for fast-agent

### Risk 4: Performance Degradation
**Challenge:** Framework overhead may slow down responses.

**Likelihood:** Low
**Impact:** Medium - User experience

**Mitigation:**
- Performance testing in Phase 5
- Benchmark before and after migration
- Optimize hot paths if needed
- Use framework's performance features

**Contingency:** If performance issues:
- Profile to identify bottlenecks
- Optimize custom code
- Consider framework configuration tuning

### ToolRunnerHooks Implementation
```python
from fast_agent.types import ToolRunnerHooks, ToolRunnerHookCapable

class ChatPlatformToolAgent(ToolAgent, ToolRunnerHookCapable):
    def __init__(self, platform_interface, approval_rules):
        self.platform = platform_interface
        self.approval_rules = approval_rules
        self._hooks = ToolRunnerHooks(
            before_tool_call=self._handle_tool_approval,
            after_tool_call=self._report_tool_result
        )
    
    @property
    def tool_runner_hooks(self):
        return self._hooks
    
    async def _handle_tool_approval(self, runner, tool_call):
        await self.platform.send_status(f"🔧 Using tool: {tool_call.name}")
        if self.approval_rules.requires_approval(tool_call.name):
            approved = await self.platform.request_approval(
                tool_name=tool_call.name,
                parameters=tool_call.input
            )
            if not approved:
                raise ToolApprovalDenied(f"User denied: {tool_call.name}")
    
    async def _report_tool_result(self, runner, message):
        if message.tool_results:
            for tool_name, result in message.tool_results.items():
                await self.platform.send_status(f"✅ {tool_name} completed")
```

### Platform Interface Specification
```python
from abc import ABC, abstractmethod

class PlatformInterface(ABC):
    @abstractmethod
    async def send_message(self, thread_id: str, content: str) -> None:
        """Send a message to the platform."""
        pass
    
    @abstractmethod
    async def send_status(self, thread_id: str, status: str) -> None:
        """Send a status update (tool usage, results, etc.)."""
        pass
    
    @abstractmethod
    async def request_approval(
        self, 
        thread_id: str, 
        tool_name: str, 
        parameters: dict
    ) -> bool:
        """Request tool approval from user. Returns True if approved."""
        pass
    
    @abstractmethod
    async def send_error(self, thread_id: str, error: str) -> None:
        """Send an error message to the platform."""
        pass
```

#### approval_rules.yaml
```yaml
# Tool-specific approval requirements
tools:
  # Filesystem tools
  filesystem.delete_directory:
    requires_approval: true
  filesystem.write_file:
    requires_approval: true
  filesystem.read_file:
    requires_approval: false
  
  # Drupal tools
  drupal.enable_module:
    requires_approval: true
  drupal.clear_cache:
    requires_approval: false
  
  # Default behavior
  default:
    requires_approval: false
```
